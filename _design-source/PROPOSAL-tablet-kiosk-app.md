# 提案：店舗端末（タブレット）専用アプリ化

作成: 2026-06-12 / ステータス: **案（未着手・承認待ち）** / 対象: NOXA 店舗端末モード

> 「デバイスパスワードでログインして使うタブレット用 UI（席回し / POS / 初回案内 / 勤怠）」を、Web のままではなく**店舗据置タブレット専用アプリ**として切り出す提案。今すぐ作らず、方針だけ確定しておく。

---

## 1. なぜアプリ化するか（Web のままの限界）

現状の店舗端末モードは Web（`AccountShell` の `device.isDevice` 分岐＋`device_profiles/panel` のパスワードログイン）。これをブラウザで運用すると次が弱い：

1. **オフライン会計が落ちる**（最重要）。POS の会計は `runTransaction`＝オンライン必須で、回線断だと失敗する。Firestore の `persistentLocalCache` を入れても**トランザクションはオフライン非対応**。電波の悪い店舗で会計が止まる。
2. **キオスク固定ができない**。ブラウザのタブ閉じ・誤操作・別サイト遷移・iOS Safari のストレージ自動削除。据置端末として事故りやすい。
3. **レシート印刷・キャッシュドロア等の周辺機器**に繋げない（将来の「レシート認識/印刷」要件と直結）。
4. 起動が「ブラウザ→URL→ログイン」で、現場スタッフには非自明。

→ 据置タブレットは**アプリの方が筋が良い**。オフライン・キオスク・ハードウェアが native で解ける。

---

## 2. 対象スコープ（店舗端末アプリに載せるもの）

スタッフが共有タブレットで触る**現場運営**だけ。個人機能・契約・公開プロフィール等は載せない。

- **席回し**（フロアボード／卓タップ→詳細）
- **POS（会計）**（席回しに統合済みの伝票・会計）
- **初回案内**（指名タブレット）
- **勤怠**（出退勤打刻）
- ログイン＝**デバイスパスワード**（`device_profiles/panel` の PIN → claims.shopId）。個人ログイン不要。

載せないもの：給与/売掛/リスク客（既に device で非表示）、個人機能、マイページ、連携・契約、公開プロフィール。

---

## 3. アプローチ比較

| 案 | 内容 | 既存資産流用 | オフライン | キオスク/ハード | 工数 |
|---|---|---|---|---|---|
| **A. Capacitor ラップ（推奨）** | 既存 Next.js の店舗端末画面を Capacitor で iOS/Android アプリ化 | ◎ ほぼ全流用 | ○（要設計：後述） | ◎ | 中 |
| B. PWA インストール | manifest＋SW でホーム追加（実装済みの延長） | ◎ | △（SWのみ・tx不可は同じ） | △（iOSは弱い） | 小 |
| C. フルネイティブ別実装（SwiftUI/Compose） | 端末アプリを別途新規実装 | ✕ | ◎ | ◎ | 大 |

**推奨＝A（Capacitor）**。理由：既に作った Web の店舗端末 UI（席回し/POS/初回案内/勤怠）をそのまま資産として使え、native の殻でオフライン・キオスク・将来のレシート印刷を足せる。host-menu-app（Capacitor Android）/ yorulog-ios の前例もあり。

---

## 4. 技術構成（案 A の詳細）

- **シェル**：Capacitor（iOS/Android）。`webDir` に NOXA のビルド成果（店舗端末向けに `/seating` を起点とする薄いエントリ）。または既存 Vercel をラップしつつ要所だけ native plugin。
- **ログイン**：起動時に保存済みデバイストークン（Secure Storage）→ 無ければ PIN 入力 →（既存の device claims フロー / Cloud Functions）。`account/login` の個人ログインは出さない。
- **オフライン会計（要設計・ここが肝）**：
  - 案1：会計を `runTransaction` から **batched write＋冪等キー（slipId ベース）＋ローカルキュー**に再設計し、オフライン投入→復帰時フラッシュ。卓伝票はローカル DB（SQLite/Capacitor Preferences）に持ち、`seating_tables` 同期は復帰時マージ。
  - 案2：会計は当面オンライン必須のまま、native で**接続状態を明示**＋失敗時リトライキュー UI。
  - → まず案2で安全に出し、案1へ移行が現実的。
- **キオスク**：Android = Lock Task Mode（単一アプリ固定）、iOS = Guided Access / MDM。ナビは下部タブ（席回し/会計/初回案内/勤怠）のみ。
- **周辺機器（将来）**：Bluetooth/USB サーマルプリンタ（レシート）、「レシート認識」はカメラ＋OCR。native plugin で対応。
- **更新**：Web 資産は OTA（Capacitor Live Update 等）で即時反映、native 殻は store 配布。

---

## 5. フェーズ

1. **Phase 0（方針確定）**：本提案の承認＋オフライン会計を案1/案2どちらで出すか決定。
2. **Phase 1（ラップ）**：Capacitor プロジェクト作成、店舗端末画面（席回し/POS/初回案内/勤怠）をラップ、PIN ログイン、キオスク固定。＝「アプリで起動して現場が使える」状態。
3. **Phase 2（オフライン堅牢化）**：会計のローカルキュー＋冪等同期（案1）。接続状態 UI。
4. **Phase 3（ハード）**：レシート印刷 → レシート認識（OCR）。

---

## 6. 既知の論点（先に決めておくと速い）

- **オフライン会計＝tx を捨てるか**：現状 tx は「同時編集の整合」も担保している。オフライン対応＝冪等キー＋マージ設計に置き換える必要があり、ここが一番の設計コスト。
- **個人アプリ（yorulog-ios）との棲み分け**：本アプリは「店舗の共有端末」専用。個人スタッフの売上/顧客は yorulog アプリ側。役割を混ぜない。
- **データは noxa-platform 共有のまま**：アプリ化してもバックエンドは同じ Firestore。UI と同期層だけ native 化。

---

## 一言

据置タブレットは Web より**アプリが正解**（特にオフライン会計とキオスク）。ただし「会計のオフライン化」が設計の山なので、**まず Capacitor ラップ＋接続状態の明示（案2）で出し、後からローカルキュー（案1）**が安全。今は本案を承認スコープとして寝かせ、着手時にこの順で進める。
